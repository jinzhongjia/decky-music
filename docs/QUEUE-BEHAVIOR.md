# Steam Deck 音乐插件：播放队列（Queue）行为设计规范

本规范定义了 `decky-music` 插件在不同场景下触发播放时，播放队列（Queue）的语义表现。队列的“真相源”由后端的 Python bridge 维护，前端 UI 只负责触发动作和展示状态。

## 1. 核心队列模式

队列在架构上分为两种互斥的模式：**普通列表模式（Normal Queue）** 与 **电台流模式（Radio Stream）**。

### 1.1 普通列表模式 (Normal Queue)

- **特征**：有限长度的静态列表（如歌单、专辑、搜索结果），用户可查看完整队列、手动指定上一首/下一首。
- **循环策略**：支持单曲循环（Single Loop）、列表循环（List Loop）、随机播放（Shuffle）。
- **持久化**：随 `settings.json` 落盘，重启 Steam Deck 后可恢复播放。

### 1.1.1 普通队列的持久化细则

普通列表模式的队列随 `settings.json` 落盘，结构为 `"queue": {"ids": [...], "index": 3}`。只存 songId 列表与当前索引，**绝不存解析出的播放 URL**（限时签名会过期），恢复时按 id 重新向 provider 解析。配合原子写（临时文件 + `os.replace()`），Steam Deck 重启或插件重载后队列与播放位置均可恢复。

### 1.2 电台流模式 (Radio Stream)

- **特征**：动态、无限或分批拉取的歌曲流（如网易云私人 FM、QQ 猜你喜欢、QQ 雷达推荐）。
- **行为限制**：
  - **无「上一首」概念**：电台流只能往前走，不支持回退。
  - **无「播放模式」切换**：强制为顺序流，单曲循环或随机在电台模式下无意义。
  - **动态补水（Hydration）**：当当前批次播放到倒数第二首时，bridge 自动向 provider 请求下一批数据并追加到内存队列。
- **UI 表现**：隐藏或禁用迷你播放条的「上一首」和「播放模式」按钮；播放队列面板显示为“电台模式，不展示完整队列”。

## 2. 播放触发场景与队列策略矩阵

当用户在不同页面按下 `A` 键（播放）或 `X` 键（菜单）时，队列的替换与追加策略如下：

| 触发场景 | 用户操作 | 队列行为策略 | 模式状态 |
| :--- | :--- | :--- | :--- |
| **歌单/专辑/排行榜** | 点击「播放全部」 | **全量替换**：清空当前队列，将该列表所有歌曲覆盖写入，并从第 1 首开始播放。 | 普通列表 |
| **歌单/专辑/排行榜** | 点击列表中某首单曲 | **上下文替换**：清空当前队列，将**该单曲所在的整个列表**写入队列，并将播放索引（Index）定位到该单曲。 | 普通列表 |
| **搜索结果页** | 点击某首单曲 | **全量替换**：清空当前队列，将当前页的搜索结果列表写入队列，从选中曲目开始播放。 | 普通列表 |
| **任意列表单曲** | `X` 键 ->「下一首播放」 | **插队追加**：不改变当前队列主体，仅将该单曲插入到当前播放索引之后（Index + 1）。 | 普通列表 |
| **任意列表单曲** | `X` 键 ->「添加到队列末尾」 | **尾部追加**：不改变当前队列主体，将该单曲追加到队列最后。 | 普通列表 |
| **智能电台/私人 FM** | 点击「猜你喜欢/雷达/FM」入口 | **模式覆写**：清空当前普通队列，**切换为电台流模式**，拉取第一批数据并开播。 | 电台流 |

### 2.1 模式切换的互斥规则

- **从普通列表 -> 电台流**：原普通列表被彻底抛弃（不保存现场），进入电台流。
- **从电台流 -> 普通列表**：电台流被丢弃，重新建立静态列表。
- **跨 Provider 切换**：若当前在 QQ 音乐播放，用户通过 QAM 切换到网易云音乐，**当前队列被强制清空**（两个 provider 的 Song ID 体系互不通用：QQ 为 `mid`/`media_mid`，NCM 为数字 `id`），播放立即停止，迷你播放条进入空态。
- **登录态不随切换清除**：`settings.json` 中 `accounts` 按 provider 分 key 存储（`{"qq": "<cookie>", "ncm": "<cookie>"}`），两份 cookie 独立同时持有。切换 provider 只决定 bridge 在 spawn 对应 provider 进程时注入哪一份，来回切换无需重新扫码，除非 cookie 自然过期（QQ 侧可用 `refresh_credential` 静默续期）。

## 3. Bridge 侧队列控制机制

由于队列状态完全归属于 Python bridge（见 `DESIGN.md`），bridge 必须处理 `player` 的 `ended` 事件以决定下一首行为。

### 3.1 自动切歌（Auto-Advance）逻辑

当 bridge 收到 player 的 `ended` 事件（协议 v1：`{"ev":"player","type":"ended","data":{}}`，即 `protocol.ChildEvent(ev="player", type="ended")`，见 AGENTS.md「协议 v1」）：

1. **单曲循环**：`seek(0)` 重新播放当前索引。
2. **普通列表**：索引 `+1`。若到达末尾：
   - 列表循环：索引重置为 `0`，播放第一首。
   - 随机播放：从随机洗牌后的内部索引中取下一首。
3. **电台流模式**：索引 `+1`。若当前已是内存批次的倒数第二首，触发异步任务调用 `provider` 的对应接口（如 `get_guess_recommend`）拉取新数据并追加到队列。

### 3.2 队列为空（Empty State）

- **触发**：初次安装插件、跨 Provider 切换后、或用户手动清空队列。
- **表现**：迷你播放条进入空态（封面置灰、标题显示“暂无播放”、进度条禁用）。
- **拦截**：此时按 `Start` 键（盲操播放）无效。

## 4. UI 队列面板 (Y 键抽屉)

在普通大屏页面或非 provider 专属电台页面按 `Y` 键，将从右侧滑出当前播放队列面板。
QQ 智能电台、NCM 私人 FM 等专属沉浸页可按各自 UI 规格覆盖 `Y` 键（如红心），队列入口改由页面按钮或播放条提供。

- **普通模式下**：
  - 列表渲染当前队列的所有歌曲。
  - 当前播放的歌曲行高亮，并带有类似声波跳动的图标。
  - 支持在队列面板内按 `X` 键唤出菜单：移除该曲、清空队列。
- **电台模式下**：
  - 队列面板**不展示未来曲目**（保持电台的未知感）。
  - 仅展示当前正在播放的单曲卡片，并配以文案：“正在收听智能电台”。
  - 提供一个按钮：“退出电台模式并清空”。

## 5. 持久化总览

| 状态 | 是否持久化 | 说明 |
| :--- | :--- | :--- |
| 普通队列（ids + index） | 是 | 随 `settings.json` 落盘，重启恢复；只存 id，URL 重新解析 |
| 电台流内容 | 否 | 动态推荐流存了无意义；只记 `queue_mode: "radio"` 与电台类型，重启后迷你播放条显示「继续收听雷达推荐」入口，用户确认后重新开流 |
| 播放模式 / 音量 | 是 | bridge 统一持有 |
| 登录态（双 provider cookie） | 是 | 按 provider 分 key，切换 provider 不清除，`chmod 0600` 私有 |
| 跨 provider 切换后的队列 | 清空 | Song ID 体系互不通用，无保留意义 |
