# 待验证登记(pending-verify issue)

本项目 `AGENTS.md` 的 Testing rules 要求:含运行时/UI 行为的改动要在真机(gamescope 会话)
验收后才算数。但常有验证被阻塞的情况 —— 没拿到部署授权、或用户明确要"先提交后验证"。

这时**不要把改动说成"已完成/已验证"**。开一个 issue 把"还差哪几步验证"钉下来,
让它可追踪、不丢。本 skill 是"登记"这一半;真机验证怎么做见 **`decky-dev` skill**。

## 何时用

同时满足:

1. 改动落在需要真机验收的面上 —— player 出声、UI 视觉/文案/布局/手柄焦点、
   注错误/杀后端/畸形数据/断网的纵深、日志内容。
2. 现在**没法**当场在真机验完 —— 未获部署授权(见 AGENTS.md「Agent behavior:重新部署前需用户同意」),
   或用户要求先提交/推送。

只改文档、纯内部重构无运行时表现、或已当场真机验完的,**不用**开。

## 怎么做

1. 先确认代码已提交(拿到 commit 短 hash)。issue 必须能指回具体提交。
2. `gh issue create`,body 用下面的模板。验收清单**逐条映射** AGENTS.md Testing rules,
   只留本次改动真正涉及的项,别塞不相关的。
3. 回话里给出 issue 链接,并明确说明"已提交、真机验证待做(见 #N)",不得描述为已验收。
4. 验证完成后把清单勾满、附真机截图,再关 issue。UI 视觉变更还要按 AGENTS.md 更新
   `docs/ui-design/assets/device-screenshots/` 并提示重渲染。

## 模板

```bash
gh issue create --title "待验证:<一句话说清验的是什么>(真机)" --body "$(cat <<'EOF'
关联提交:<shorthash> `<commit subject>`

## 背景
<改了什么、为什么、修复前后的行为差异。已过的检查(单测/tsc/lint)也写上,并点明"未在真机验证"。>

## 待验证(真机 / gamescope 会话)
<按需保留,删掉不相关项>
- [ ] 复现触发条件:<怎么制造这个场景>
- [ ] player 出声正常 / `ldd` 只动态依赖 libasound(涉及音频时)
- [ ] 目标 UI 出现预期表现(中/英各看一次)
- [ ] 注错误 / 杀后端 / 畸形数据 / 断网:Steam UI 不崩不冻,手柄焦点正常
- [ ] 日志符合预期,且不含任何敏感信息(URL/cookie/凭证)
- [ ] 恢复正常态后行为回归

## 复现办法
<制造该场景的具体步骤>

## 验收标准
上述全部勾选;UI 变更附真机截图。验证完关闭本 issue。
EOF
)"
```

## 红线

- issue body 同样遵守日志红线:**不写任何 URL(限时 token)/ cookie / 凭证**。
- 在验证真正做完前,别在提交信息、PR、回话里把改动写成"已验证/已验收"。
- issue 只是"欠条",不是替代验证。能当场验就当场验,别为了省事而开 issue 拖着。
